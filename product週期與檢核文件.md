#product週期與檢核文件

###一、product週期總覽
本產品採用以下 Release 週期：
- **Release 頻率**：☐ 每週 ☐ 每兩週 ☐ 每月 ☑ 不固定（說明：功能完成即可發布）
- **每次 Release 原則**：
  - ☑ 小步快跑（少量變更）
  - ☑ 可回滾（使用 Git 版本控制）
  - ☑ 不影響既有核心功能

---

## 二、Release 各階段定義與檢核

### 階段 1：需求與規劃（Planning）

**目標**：讓使用者可以用網頁查看實習紀錄

**檢核問題**：
- **使用者定義**：自己、主管
- **使用情境**：
  - 日常記錄實習心得
  - 週會/月會報告準備
  - 實習總結與回顧
  - 面試作品集展示
  
- **功能需求**：
  - 一筆實習紀錄包含哪些欄位？**標題、內文、標籤、日期**
  - 頁面會顯示哪些實習紀錄欄位？**標題、內文、標籤、建立時間、更新時間**
  - 需要哪些操作功能？**新增、查看、編輯、刪除、搜尋、排序**

- **非功能需求**：
  - 資料儲存方式：LocalStorage（瀏覽器本地儲存）
  - 效能要求：頁面載入時間 < 2 秒
  - 使用者體驗：簡潔、直覺、易用

- **已知限制**：
  - 資料僅存於本地瀏覽器，清除快取會遺失
  - 不支援多裝置同步
  - 不支援圖片上傳

---

### 階段 2：開發（Development）

**目標**：功能完成並可部署

**檢核問題**：
- ☑ 是否所有需求皆有對應實作？（使用 Antigravity 製作網頁）
- ☑ 是否避免未規劃功能混入？
- ☑ 程式碼是否有適當註解？
- ☑ 是否有基本的錯誤處理機制？
- ☑ CSS 樣式是否一致？
- ☑ 是否符合響應式設計（RWD）？
- ☑ 是否有考慮資料驗證（表單驗證）？

**開發完成清單**：
- ☑ 首頁（index.html）- 紀錄列表、搜尋、排序
- ☑ 詳細頁（record-detail.html）- 完整紀錄內容、編輯/刪除
- ☑ 新增/編輯頁（add-record.html）- 表單輸入
- ☑ 資料管理模組（common.js）- CRUD 操作
- ☑ 樣式系統（styles.css）- 一致的設計語言

---

### 階段 3：測試（Testing）

**目標**：確認功能符合需求且不影響系統

**檢核問題（Web 專用）**：
- ☑ 若未撰寫單元測試，是否已由開發者自行驗證功能正確？
- ☑ 主要頁面是否可正常載入？
- ☑ 是否已確認核心功能可正常使用？
- ☑ 表單是否正常運作？
- ☐ 瀏覽器相容性是否確認？（Chrome, Edge, Safari）

**功能測試清單**：
1. **新增紀錄**
   - ☑ 可以成功新增紀錄
   - ☑ 必填欄位驗證正常
   - ☑ 標籤可以正確解析（逗號分隔）
   - ☑ 新增後自動跳轉至首頁

2. **查看紀錄**
   - ☑ 首頁正確顯示所有紀錄
   - ☑ 顯示標題、內文摘要、標籤、建立時間、更新時間
   - ☑ 點擊可進入詳細頁
   - ☑ 詳細頁顯示完整內容

3. **編輯紀錄**
   - ☑ 可以正確載入既有資料
   - ☑ 修改後成功儲存
   - ☑ 更新時間正確記錄
   - ☑ 編輯後返回詳細頁

4. **刪除紀錄**
   - ☑ 刪除前有確認提示
   - ☑ 刪除後正確移除資料
   - ☑ 刪除後跳轉至首頁

5. **搜尋功能**
   - ☑ 可以搜尋標題、內文、標籤
   - ☑ 即時顯示搜尋結果
   - ☑ 搜尋結果高亮顯示關鍵字

6. **排序功能**
   - ☑ 可以按日期排序（最新/最舊）
   - ☑ 可以按標題排序

7. **資料驗證**
   - ☑ 空白標題無法送出
   - ☑ 空白內文無法送出
   - ☑ 日期格式正確

8. **使用者體驗**
   - ☑ 操作有適當的提示訊息
   - ☑ 頁面切換流暢
   - ☑ 動畫效果正常

### 階段 4：正式 Release（Production）

**目標**：正式對使用者提供服務

**檢核問題（Web 專用）**：
- ☐ 使用者是否能在環境正常開啟並瀏覽所有必要頁面？
- ☐ 實際使用者是否能依需求完成核心操作流程？
- ☐ 是否已部署至可存取的環境？（本地/線上）
- ☐ 是否有提供使用說明或教學？

**部署檢核**：
- ☐ 所有檔案路徑正確
- ☐ 圖片/資源檔案可正常載入
- ☐ JavaScript 功能正常運作
- ☐ CSS 樣式正確顯示
- ☐ 在實際使用環境測試通過

---

### 階段 5：Release 後監控與維護（Post-release）

**目標**：即時發現問題並快速處理，持續優化

**監控檢核**：
- 是否有異常錯誤率上升？（開發者工具 Console 檢查）
- 是否有使用者回報阻斷性問題？
- 資料是否正常儲存？
- 效能是否符合預期？

**定期維護**：
- ☐ 每週/每月檢視使用狀況
- ☐ 定期備份資料（LocalStorage 匯出）
- ☐ 記錄使用中發現的問題
- ☐ 收集功能改進建議

**優化計畫**：
- 是否有需要改善的使用體驗？
- 是否需要新增功能？
- 是否需要效能優化？
- 文件是否需要更新？

## 四、版本控制與回滾機制

**版本控制策略**：
- 使用 Git 進行版本控制
- 每次功能完成建立一個 commit
- 重要版本建立 tag（例如：v1.0, v1.1）

**回滾步驟**：
1. 確認問題嚴重程度
2. 使用 `git log` 查看版本歷史
3. 使用 `git revert` 或 `git reset` 回到穩定版本
4. 重新部署
5. 通知使用者

**資料備份**：
- 定期匯出 LocalStorage 資料
- 保存重要紀錄的備份檔案
- 建議使用者自行備份
